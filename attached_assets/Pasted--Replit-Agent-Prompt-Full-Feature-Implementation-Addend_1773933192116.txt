This is the third and final addendum. Apply all changes on top of the previous two prompts. Do not re-implement anything already built. All existing tabs, routes, pages, services, and DB models must continue to work exactly as before.

## DATA SOURCES DECISION

**Stocks news**: Use **Finnhub** (`https://finnhub.io/api/v1`) — free tier provides:

- `GET /company-news?symbol={symbol}&from={date}&to={date}` — real company news with headline, summary, source, datetime, url
- `GET /news-sentiment?symbol={symbol}` — pre-computed sentiment score, buzz score, article count
- `GET /calendar/earnings?from={date}&to={date}` — earnings calendar with EPS estimates, surprise data
- `GET /stock/recommendation?symbol={symbol}` — analyst buy/sell/hold consensus
- `GET /stock/upgrade-downgrade?symbol={symbol}` — analyst upgrades/downgrades with price targets
- Auth: `?token=FINNHUB_API_KEY` query param or `X-Finnhub-Token` header
- Free tier rate limit: 60 calls/minute — implement a rate-limit queue

**Crypto news**: Use **CryptoPanic** (`https://cryptopanic.com/api/free/v1/`) — free tier:

- `GET /posts/?auth_token={token}&currencies={BTC,ETH}&public=true&filter=important` — real crypto news with votes, sentiment filter
- `filter` options: `rising` | `hot` | `bullish` | `bearish` | `important` | `saved` | `lol`
- Free tier: no pagination limit for basic posts, ~200 req/hour
- Auth: `auth_token=CRYPTOPANIC_API_KEY` query param

**Sector ETF data for sector rotation**: Use Yahoo Finance (already integrated) — fetch SPY, XLK, XLF, XLE, XLV, XLI, XLY, XLP, XLRE, XLU, XLC daily performance. No additional key needed.

**Live P&L WebSocket for stocks**: Use Finnhub WebSocket `wss://ws.finnhub.io?token={FINNHUB_API_KEY}` — subscribe with `{"type":"subscribe","symbol":"AAPL"}` to get real-time trade prices. For crypto live P&L, use the existing Hyperliquid WebSocket infrastructure already in the codebase.

**Secrets to add** (user must set these in Replit Secrets):

- `FINNHUB_API_KEY` — from finnhub.io free account
- `CRYPTOPANIC_API_KEY` — from cryptopanic.com free account
- `TELEGRAM_BOT_TOKEN` — from @BotFather on Telegram
- `TELEGRAM_CHAT_ID` — populated automatically via setup flow (see Phase B)

### 1A. Finnhub News Provider

**File**: `server/src/services/news/FinnhubNewsProvider.ts`

Replace `generateNewsItems()` in `generator.ts` with real Finnhub data. Create a provider that:

- `fetchCompanyNews(ticker: string, daysBack = 5): Promise<NormalizedNewsItem[]>`:

### 4B. Journal Service

**File**: `server/src/services/journal/TradeJournalService.ts`

**`generatePostMortem(closedPositionId: string, userId: string): Promise<TradeJournal>`**

1. Load the `ClosedPosition` record — entry/exit price, hold days, thesis, close reason, outcome, stop loss, target
1. Load the `AgentOutput` records for this ticker around the time of entry (use `createdAt` closest to `openedAt`)

   Hold time: {days} days | Close reason: {closeReason}

   Invalidation was: ${stopLoss}

     "exitQuality": "DISCIPLINED|TOO_EARLY|TOO_LATE|PANIC",

1. Parse the JSON response and persist to `TradeJournal`

## PHASE 6 — Sharpe & Sortino Ratios

In `server/src/services/analytics/PerformanceService.ts`, add to `getOverview()`:

```typescript
// Add to the return object of getOverview():
sharpeRatio: number,   // (avgReturn - riskFreeRate) / stdDevReturns, annualized
sortinoRatio: number,  // (avgReturn - riskFreeRate) / downsideDeviation, annualized
calmarRatio: number,   // annualizedReturn / maxDrawdownPct
```

Implementation:

- `riskFreeRate`: use 0.045 / 252 as daily risk-free rate (approx 4.5% annual T-bill rate)
- `stdDevReturns`: standard deviation of daily `pnlPercent` values across all trades
- `downsideDeviation`: standard deviation of only negative `pnlPercent` values (Sortino uses downside only)
- `annualizedSharpe = (avgDailyReturn - dailyRiskFree) / stdDev * sqrt(252)`
- `annualizedSortino = (avgDailyReturn - dailyRiskFree) / downsideDev * sqrt(252)`
- `maxDrawdownPct`: compute the maximum peak-to-trough decline in the equity curve
- `calmarRatio = annualizedReturn / maxDrawdownPct` (handle divide-by-zero)

In `PerformanceLab.tsx`, add to the Overview tab’s stat cards row (currently shows winRate, totalPnl, etc.):

- Sharpe Ratio card: value colored green if >1.5, yellow if 0.5–1.5, red if <0.5. Tooltip: “Risk-adjusted return. >1.5 is excellent, >1 is good.”
- Sortino Ratio card: same color thresholds. Tooltip: “Like Sharpe but only penalizes downside volatility.”
- Calmar Ratio card: green if >1, yellow 0.5–1, red <0.5. Tooltip: “Annual return divided by max drawdown.”

## PHASE 11 — Multi-Timeframe Intraday Scan

In `server/src/services/scans/dailyScanOrchestrator.ts`, add a new scan mode:

**`runIntradayConfirmationScan(topTickers: string[]): Promise<IntradayConfirmation[]>`**:

- Takes the top 10 tickers from the most recent daily scan run

## PHASE 12 — Paper → Live Graduation

**File**: `server/src/services/autotrader/GraduationService.ts`

```typescript
interface GraduationEligibility {
  positionId: string;
  ticker: string;
  eligible: boolean;
  reasons: string[];
  warnings: string[];
  suggestedExchange: 'tos' | 'hyperliquid';
  suggestedSize: number;
}
```

**`checkGraduationEligibility(positionId: string): Promise<GraduationEligibility>`**:

- Load the `PaperPosition` and its `PositionSnapshot` history
- Eligible criteria:
  - Position has been open ≥ 3 days
  - Thesis health has stayed above 50 for all snapshots
  - Current thesis health ≥ 60
  - No `SETUP_INVALIDATED` or `INVALIDATION_THREATENED` alerts in the history
  - Thesis bias is still aligned with the original entry bias
  - The thesis conviction score from the latest analysis is ≥ 65
- Warnings (not blocking): earnings within 7 days, high volatility, position near resistance

**`graduateToLive(positionId: string, userId: string): Promise<GraduationResult>`**:

- Calls `checkGraduationEligibility` — throws if not eligible
- Determines exchange (crypto → HL, stock → TOS)

- Places the live order via the appropriate exchange service
- Logs to `AutoTradeLog` with action `'GRADUATE'`
- Sends Telegram notification: “🎓 Paper position graduated to live: {ticker} → {exchange}”

### 13B. Backtest Service

**File**: `server/src/services/backtest/BacktestService.ts`

**`runBacktest(params: BacktestParams): Promise<string>`** (returns backtest run ID, runs async):

```typescript
interface BacktestParams {
  userId: string;
  name: string;
  startDate: string;  // 'YYYY-MM-DD'
  endDate: string;
  assetScope: 'ALL' | 'STOCKS_ONLY' | 'CRYPTO_ONLY';
  config: ExchangeAutoConfig;  // the parameter set to test
}
```

Implementation:

1. Create a `BacktestRun` DB record with status `'pending'`
1. Run asynchronously (don’t await in the route handler):
1. Load `DailyScanResult` records from the DB between `startDate` and `endDate` — these are historical scan results already persisted from past runs
1. For each scan result date in chronological order:
- Apply the config filters (minConvictionScore, allowedBias, etc.)
- For each passing candidate: fetch historical OHLCV for the entry date from Yahoo Finance
- Simulate entry at `entryZone.low` or next day open (use actual open price from historical bars)
- Track position using historical daily closes:
  - If close < stopLoss: exit at next day open, record as loss
  - If close > takeProfit1: exit at next day open, record as win
  - If hold exceeds `suggestedHoldWindow` days: exit at close, record result
- Apply `PositionSizer` logic using simulated equity (starting at $100,000)
1. Compute all performance metrics: win rate, total return, Sharpe, Sortino, max drawdown, profit factor, avg hold days
1. Save results to `BacktestRun.results` and update status to `'completed'`

**Limitation note in code comments**: “Backtest quality is limited by available historical scan data. Results are only meaningful after the platform has been running for 30+ days and has accumulated scan history.”
