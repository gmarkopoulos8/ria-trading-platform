# Replit Agent Prompt — Full Market Universe Scanner

This is an addendum to the previous prompts. Apply on top of everything already built. Do not re-implement, rename, or remove any existing files, models, routes, or services. Extend only.

-----

## WHAT THIS BUILDS

The Opportunity Scanner currently scans a hardcoded list of ~34 tickers. This replaces that with:

1. **A Finnhub-powered pre-screener** that pulls the entire NYSE/NASDAQ common stock list (~8,000 symbols) and all major crypto from Hyperliquid, applies fast liquid filters to cut it to ~200–400 tradable candidates, then runs the full thesis engine on those candidates
1. **A two-phase scan architecture** — Phase 1 is a fast bulk filter using Finnhub quote data (price, volume, market cap). Phase 2 is deep thesis analysis on the survivors only. This keeps runtime under 20 minutes on the free Finnhub tier
1. **Intelligent rate limiting** that stays safely within Finnhub’s 60 calls/minute free limit with automatic backoff
1. **A new `FullUniverseScanner` service** that replaces the static lists in `scanUniverseService.ts` without breaking the existing `buildUniverse()` function (which still works as a fallback)
1. **A new `FinnhubDataProvider`** for stock quote/history/profile data that the scanner uses instead of Yahoo Finance during bulk scans (Yahoo Finance blocks bulk requests with 429s)

## PHASE 1 — Finnhub Data Provider

**File**: `server/src/services/market/stocks/FinnhubProvider.ts`

This provider implements `IStocksProvider` and is used **exclusively by the scanner** for bulk data fetching. Yahoo Finance remains the primary provider for individual quote lookups and chart history in the UI.

```typescript
import axios from 'axios';
import { IStocksProvider, NormalizedQuote, OHLCVBar, SearchResult, Timeframe } from '../types';

const FINNHUB_BASE = 'https://finnhub.io/api/v1';
const API_KEY = process.env.FINNHUB_API_KEY ?? '';

// Token bucket: 55 calls/minute (safe under the 60/min free limit)
// Shared across the entire module — all methods draw from the same bucket
class RateLimiter {
  private tokens: number;
  private lastRefill: number;
  private readonly maxTokens = 55;
  private readonly refillIntervalMs = 60_000;

  constructor() {
    this.tokens = this.maxTokens;
    this.lastRefill = Date.now();
  }

  async waitForToken(): Promise<void> {
    const now = Date.now();
    const elapsed = now - this.lastRefill;
    if (elapsed >= this.refillIntervalMs) {
      this.tokens = this.maxTokens;
      this.lastRefill = now;
    }

    if (this.tokens > 0) {
      this.tokens--;
      return;
    }

    // Wait for refill
    const waitMs = this.refillIntervalMs - elapsed + 100; // +100ms buffer
    console.log(`[FinnhubProvider] Rate limit reached — waiting ${waitMs}ms`);
    await new Promise((r) => setTimeout(r, waitMs));
    this.tokens = this.maxTokens - 1;
    this.lastRefill = Date.now();
  }
}

export const finnhubRateLimiter = new RateLimiter(); // exported so other services share it
```

Implement these methods on `FinnhubProvider`:

**`getStockSymbols(exchange: 'US' | 'NYSE' | 'NASDAQ' = 'US'): Promise<FinnhubSymbol[]>`**

- `GET /stock/symbol?exchange=US&token={key}`
- Returns array of `{ description, displaySymbol, symbol, type }` — filter to `type === 'Common Stock'` only
- Cache result for 24 hours (this list changes slowly) using a module-level Map
- This is a single API call for ~8,000 symbols — very efficient

**`getQuote(symbol: string): Promise<FinnhubQuote | null>`**

- `GET /quote?symbol={symbol}&token={key}`
- Returns `{ c: currentPrice, h: dayHigh, l: dayLow, o: dayOpen, pc: prevClose, v: volume, t: timestamp }`
- Always call `await finnhubRateLimiter.waitForToken()` before each HTTP request
- Return null (not throw) on any error — scanner must never crash on a bad ticker

**`getBasicFinancials(symbol: string): Promise<FinnhubMetric | null>`**

- `GET /stock/metric?symbol={symbol}&metric=all&token={key}`
- Returns market cap, 52-week high/low, avg volume, beta, P/E ratio
- Key fields from response: `metric.marketCapitalization` (in millions), `metric.52WeekHigh`, `metric.52WeekLow`, `metric.10DayAverageTradingVolume` (in millions shares/day)
- Cache per-symbol for 6 hours

**`getCandles(symbol: string, resolution: string, from: number, to: number): Promise<OHLCVBar[]>`**

- `GET /stock/candle?symbol={symbol}&resolution={resolution}&from={from}&to={to}&token={key}`
- `resolution`: `'D'` for daily, `'W'` for weekly, `'60'` for hourly
- Response: `{ c[], h[], l[], o[], v[], t[], s }` where `s === 'ok'` means success, `s === 'no_data'` means empty
- Map to `OHLCVBar[]` — return empty array if `s !== 'ok'`

**`quote(symbol: string): Promise<NormalizedQuote>`** (implements `IStocksProvider`)

- Calls `getQuote()` and maps to `NormalizedQuote`
- Needed to satisfy the interface — used when `FinnhubProvider` is injected into `ThesisEngine`

**`history(symbol: string, timeframe: Timeframe): Promise<OHLCVBar[]>`** (implements `IStocksProvider`)

- Maps `Timeframe` to `resolution` and date range:
  - `'1D'` → resolution `'5'`, from = now-1day
  - `'1W'` → resolution `'60'`, from = now-7days
  - `'1M'` → resolution `'D'`, from = now-30days
  - `'3M'` → resolution `'D'`, from = now-90days
  - `'6M'` → resolution `'D'`, from = now-180days
  - `'1Y'` → resolution `'W'`, from = now-365days
- Calls `getCandles()`

**`search(query: string): Promise<SearchResult[]>`** (implements `IStocksProvider`)

- `GET /search?q={query}&token={key}` — Finnhub symbol search
- Maps results to `SearchResult[]`

## PHASE 2 — Full Universe Scanner Service

**File**: `server/src/services/scans/FullUniverseScanner.ts`

This is the core of the feature. It runs in two phases.

### Phase 1: Symbol Fetch + Fast Filter

```typescript
export interface FilterCriteria {
  minPrice: number;           // default: 1.00 (filter penny stocks)
  maxPrice: number;           // default: 5000 (filter Berkshire-class outliers)
  minMarketCapM: number;      // default: 300 (million USD — filter micro-caps)
  minAvgVolumeM: number;      // default: 0.5 (million shares/day avg — filter illiquid)
  maxAvgVolumeM?: number;     // optional upper bound
  minPriceChangePct: number;  // default: -50 (filter halted/suspended stocks)
  maxPriceChangePct: number;  // default: 50 (filter circuit-breaker events)
  excludeExchanges: string[]; // default: ['PNK', 'OTC', 'GREY'] — OTC markets
}
```

**`fetchAndFilterStocks(criteria: FilterCriteria): Promise<CandidateAsset[]>`**:

1. Call `FinnhubProvider.getStockSymbols('US')` — get ~8,000 common stock symbols
1. Filter immediately to only `displaySymbol` values that match `/^[A-Z]{1,5}$/` — removes warrants (`.WS`), preferred shares (`.PR`), rights (`.RT`), units (`.U`), and foreign ordinary shares (`.`). This cuts to ~4,000 pure common stocks.
1. **Batch fetch metrics** in batches of 50, with rate limit spacing:
- For each symbol, call `FinnhubProvider.getBasicFinancials()`
- Apply filter: `marketCapM >= criteria.minMarketCapM` AND `avgVolume10D >= criteria.minAvgVolumeM`
- This is the most expensive step — ~80 API calls for 4,000 symbols at 50/batch. Takes ~2 minutes at 55 calls/minute.
- **Optimization**: cache the full filtered list for 4 hours. If cached, skip this entire step.
1. **Batch fetch quotes** for survivors (typically ~400–800 stocks pass the market cap + volume filter):
- Call `FinnhubProvider.getQuote()` for each
- Apply price filters: `price >= minPrice`, `price <= maxPrice`, daily change within bounds
- Deduplicate by symbol
1. Return as `CandidateAsset[]` with `assetClass: 'stock'`

**Important caching strategy**: Store the filtered candidate list in a module-level Map with a 4-hour TTL. The symbol list + metrics barely change intraday. Only quotes need to be fresh.

### Phase 2: Deep Scoring Pre-Filter

Before running the full `ThesisEngine` (which calls Yahoo Finance for history + news for each ticker), apply a **scoring pre-filter** using only Finnhub data to cut the ~400 candidates to the best ~150:

**`prescoreCandidate(symbol: string, quote: FinnhubQuote, metrics: FinnhubMetric): PreScore`**:

Compute a fast pre-score (0–100) using only already-fetched data — no additional API calls:

```typescript
interface PreScore {
  symbol: string;
  score: number;
  reasons: string[];
}
```

Score components (each 0–25, summed):

1. **Momentum score** (0–25): based on today’s price change vs average
- Change > +3%: 25 | Change +1–3%: 18 | Change +0–1%: 12 | Change -1–0%: 8 | Change < -1%: 3
1. **Volume surge score** (0–25): today’s volume vs 10-day average
- Volume ratio > 3x: 25 | 2–3x: 20 | 1.5–2x: 15 | 1–1.5x: 10 | < 1x: 5
1. **52-week position score** (0–25): where current price sits in the 52-week range
- Price in top 10% of range: 25 | top 25%: 20 | middle 50%: 12 | bottom 25%: 6 | bottom 10%: 2
- *Rationale: stocks making new highs tend to continue making new highs*
1. **Size/liquidity score** (0–25): based on market cap tier
- Large cap (>$10B): 25 | Mid cap ($2B–$10B): 20 | Small cap ($300M–$2B): 15

Keep only candidates with `prescore >= 45` (the top ~40% by this metric). This reduces the 400+ candidates to ~150, which is a manageable number for full thesis analysis.

### Phase 3: Full Thesis Analysis on Survivors

**`runFullUniverse(opts: FullScanOptions): Promise<RankedResult[]>`**:

```typescript
interface FullScanOptions {
  assetScope: 'ALL' | 'STOCKS_ONLY' | 'CRYPTO_ONLY';
  riskMode: 'ALL' | 'CONSERVATIVE' | 'AGGRESSIVE';
  filterCriteria?: Partial<FilterCriteria>;
  maxCandidates?: number;        // default: 200 (cap before thesis analysis)
  maxFinalResults?: number;      // default: 100 (top results to return)
  onProgress?: (phase: string, done: number, total: number) => void;
}
```

Implementation:

```
Phase 1: Filter stocks (if scope includes stocks)
  → fetchAndFilterStocks() → ~400 candidates → prescore → ~150 survivors
  → Emit onProgress('FILTERING', done, total) as each batch completes

Phase 2: Fetch crypto universe (if scope includes crypto)
  → Call Hyperliquid getMeta() to get ALL available perp markets
  → Call getAllMids() to get current prices  
  → Filter: only coins with mid price > $0.001, available in universe
  → Augment with CoinGecko market cap data for the top 100 by Hyperliquid OI
  → Merge with static CRYPTO_UNIVERSE from scanUniverseService (deduplicated)

Phase 3: Sort all candidates by prescore descending, take top maxCandidates

Phase 4: Run rankCandidates() from dailyRankingService with CONCURRENCY = 5
  → This calls thesisEngine.analyze() for each candidate
  → thesisEngine uses Yahoo Finance for history (already integrated)  
  → Emit onProgress('ANALYZING', done, total) continuously
  → Each thesis analysis takes ~2-4 seconds (3 API calls in parallel)
  → 150 stocks × 3s avg = ~7.5 minutes at concurrency 5

Phase 5: Return top maxFinalResults sorted by compositeScore
```

**Concurrency of 5** in the thesis analysis phase is important. Too low (3) makes it too slow. Too high (10+) will hit Yahoo Finance rate limits. 5 is the right balance.

**`getCryptoUniverse(): Promise<CandidateAsset[]>`**:

- Call `getMeta()` from `hyperliquidInfoService` — returns all ~150+ available perp markets on Hyperliquid
- Call `getAllMids()` — get current prices for all
- Filter: price > $0.001 (filters dust tokens), name not ending in `-PERP` (normalize symbol)
- Merge with the static `CRYPTO_UNIVERSE` from `scanUniverseService.ts` (keep all static ones, add new ones from HL)
- Map HL asset names to CandidateAsset format: `{ ticker: coin, name: coin, assetClass: 'crypto', isCrypto: true }`
- Deduplicate by ticker
- Return up to 80 crypto candidates (more than this slows the scan significantly with CoinGecko rate limits)

## PHASE 5 — Scan Progress WebSocket

During a full universe scan (which takes 10–20 minutes), the user needs real-time progress feedback. Add a server-sent events (SSE) endpoint — simpler than WebSocket for one-way progress streaming:

**Add to `server/src/routes/daily-scans.ts`**:

```typescript
// Store active scan progress for SSE streaming
const scanProgress = new Map<string, { phase: string; done: number; total: number; log: string[] }>();
export function updateScanProgress(scanRunId: string, phase: string, done: number, total: number, logMsg?: string) {
  const existing = scanProgress.get(scanRunId) ?? { phase, done, total, log: [] };
  existing.phase = phase;
  existing.done = done;
  existing.total = total;
  if (logMsg) existing.log = [...existing.log.slice(-20), logMsg]; // keep last 20 log lines
  scanProgress.set(scanRunId, existing);
}

// SSE endpoint
router.get('/runs/:id/progress', requireAuth, (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  res.flushHeaders();

  const send = () => {
    const progress = scanProgress.get(req.params.id);
    if (progress) {
      res.write(`data: ${JSON.stringify(progress)}\n\n`);
    }
  };

  send();
  const interval = setInterval(send, 2000); // push every 2 seconds
  req.on('close', () => clearInterval(interval));
});
```

Pass `updateScanProgress` as the `onProgress` callback in `dailyScanOrchestrator.ts`.

function getFilterCriteriaForRiskMode(riskMode: RiskMode, overrides: Partial<FilterCriteria> = {}): FilterCriteria {

    minPrice: 1.00,

    minMarketCapM: 300,

    return { ...defaults, minMarketCapM: 100, minAvgVolumeM: 0.2, ...overrides };

  return { ...defaults, ...overrides };
}

## IMPORTANT IMPLEMENTATION NOTES

**Rate limit math**: At 55 calls/minute (conservative):

- Fetching symbols: 1 call (instant)
- Fetching metrics for 4,000 symbols in batches: each `getBasicFinancials()` is 1 call → ~73 calls = ~80 seconds with rate limiting. BUT this is cached for 4 hours, so it only happens once per 4-hour window.
- Fetching quotes for ~400 survivors: ~400 calls = ~7.3 minutes. Also cached for 30 minutes.
- History for ~150 thesis analyses: 1 call each = ~2.7 minutes
- Total fresh (no cache): ~10–12 minutes. With 4-hour cache on metrics: ~4–5 minutes.

**Yahoo Finance in thesis analysis**: Each `thesisEngine.analyze()` call fetches 3M history from Yahoo Finance (1 call) plus news. Yahoo Finance has no official rate limit but blocks above ~30 req/min. With CONCURRENCY=5 and each analysis taking 2-4 seconds, the effective rate is ~75–150 req/min across all concurrent analyses — this will trigger 429s. Solutions:

- Add a 500ms delay between each batch in `rankCandidates()` when `scanMode=true`
- Or use Finnhub’s `getCandles()` instead of Yahoo Finance for history when `scanMode=true` (preferred — use the FinnhubProvider.history() method added in Phase 1)
- The Finnhub history calls count against the rate limit bucket, so at CONCURRENCY=5 with 5 history calls in-flight, add a per-call delay of `60000ms / 55 calls * 5 concurrent = 5.5s` between batches. This is handled by `finnhubRateLimiter.waitForToken()` which each call awaits individually.

**Symbol filtering regex**: `/^[A-Z]{1,5}$/` filters:

- `BRK.B` (dot = foreign/class shares) ✓ filtered out
- `AAPLW` (5 letters = warrant) — wait, actually warrants are usually `AAPLWS` or end in `.WS`. Use `/^[A-Z]{1,4}$|^[A-Z]{5}$/` and additionally filter where `type !== 'Common Stock'` per Finnhub’s own type field. Trust Finnhub’s `type === 'Common Stock'` classification as the primary filter.

**Never modify**: `buildUniverse()` in `scanUniverseService.ts`, the existing `ThesisEngine.analyze()` public signature (only add optional `opts` param), `rankCandidates()` public signature, any existing routes or models.

**Error resilience**: Every Finnhub call must be wrapped in try/catch and return a safe default (null / empty array) rather than throwing. A single bad symbol must never abort a scan of 400 symbols. Log errors at `warn` level with the symbol name for debugging.

**Memory**: Scanning 400 tickers × full thesis results × JSON = potentially 50–100MB in-memory during a scan. After persisting to DB and returning results, clear the in-memory results array. The module-level candidate cache (4-hour TTL) should only store `CandidateAsset[]` (lightweight), not full thesis results.
